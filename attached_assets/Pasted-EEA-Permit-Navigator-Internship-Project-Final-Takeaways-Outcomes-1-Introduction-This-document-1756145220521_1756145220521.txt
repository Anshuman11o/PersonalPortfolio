EEA Permit Navigator Internship Project - Final Takeaways & Outcomes
1. Introduction
This document compiles the outcomes, lessons learned, and final deliverables from my internship project under the AI for the Commonwealth (AI4CW) Cohort 3 program in collaboration with the Massachusetts Executive Office of Energy and Environmental Affairs (EEA). The project aimed to design and implement a Permit Navigator AI Chatbot to improve public access to permit information and reduce the workload on EEA support staff.
 
2. Problem Context & Motivation
Problem:
•	EEA’s current permit/licensing system is fragmented, text-heavy, and difficult to navigate.
•	Users—ranging from contractors to residents—struggle to identify the correct permits.
•	Support staff spend thousands of hours annually answering repetitive queries.
Impact of the Problem:
•	High frustration for both users and staff.
•	Delays in time-sensitive projects.
•	Increased operational costs due to manual support processes.
Why It Matters:
•	Public-facing usability is an accessibility issue.
•	Addressing this problem supports equitable access, faster service, and more efficient government operations.
 
3. Solution Overview
The solution is an AI-powered, human-like chatbot integrated into the EEA ePlace Portal to guide users to the correct permits through structured, conversational interactions.
Core Features:
•	Conversational, guided navigation with follow-up questions (query funneling).
•	Context-aware responses with direct links to relevant resources.
•	Guardrails for out-of-scope inputs.
•	Admin dashboard for monitoring, link updates, and pipeline control.
•	Accessibility-first design (WCAG compliance).
Technical Highlights:
•	Frontend: Next.js chatbot widget + admin dashboard.
•	Backend: AWS Lambda, Step Functions, S3.
•	AI/LLM: Claude 3.7 Sonnet via AWS Bedrock (RAG approach with Knowledge Base + OpenSearch).
•	Scraping Tools: BeautifulSoup (static) + Firecrawl API (dynamic).
•	Orchestration & Hosting: AWS Amplify (CI/CD), CloudFormation (IaC).
 
4. Development Journey & Phases
Phase 1 – Problem Definition & Research (Mar–Apr 2025)
•	Analyzed mass.gov permit pages, metadata, and Drupal tags.
•	Mapped EEA’s agency structure and permit categorization.
•	Reviewed Grants Navigator chatbot for limitations.
•	Created Problem Definition Memo documenting user pain points and agency workflows.
Phase 2 – Human-Centered Design (Apr–May 2025)
•	Developed empathy maps for different user personas.
•	Designed conversational flows for dynamic filtering and redirection.
•	Proposed accessibility and multilingual capabilities.
•	Delivered HCD Strategy Document.
Phase 3 – MVP Development (May–Jun 2025)
•	Compared AWS vs Azure solutions → selected AWS for scalability and fit.
•	Built autonomous data pipeline for scraping, metadata filtering, and indexing into S3 & Bedrock KB.
•	Developed first functional chatbot prototype.
Phase 4 – Iteration & Integration (Jun–Jul 2025)
•	Implemented conversational improvements with AWS Prompt Flow.
•	Connected pipeline components with AWS Step Functions.
•	Built admin dashboard prototype for pipeline management and analytics.
•	Conducted scenario-based testing with EEA staff.
Phase 5 – Finalization & Handoff (Jul–Aug 2025)
•	Completed deployment in AWS Amplify.
•	Delivered Implementation & Handoff Guide with cost estimates and operational instructions.
•	Conducted user acceptance testing; achieved improved navigation accuracy and reduced irrelevant responses.
 
5. Key Deliverables
•	Problem Definition Memo
•	Human-Centered Design Strategy
•	Technical Specification Memo (system architecture, tech stack, security, compliance)
•	MVP Prototypes (AWS-based chatbot & admin dashboard)
•	Final AI4CW Presentation
•	Implementation & Handoff Guide (setup, deployment, maintenance)
•	Cost Analysis (~$658/month for estimated traffic)
 
6. Measured Outcomes & Impact Potential
Planned KPIs:
•	25% reduction in time spent finding permits.
•	≥40% self-navigation success rate in pilot.
•	≥70% user satisfaction score.
•	20% reduction in navigation-related support tickets.
Potential Annual Staff Time Savings:
•	~5,000+ hours (15 repetitive tickets/day × 1 hr × 365 days).
Scalability:
•	Architecture supports onboarding of multiple EEA sub-agencies.
•	Expandable to licensing or other state agency services.
•	Potential multilingual support for broader accessibility.
 
7. Lessons Learned
From Development:
•	Start user testing earlier to avoid late-stage redesigns.
•	Limit over-exploration of tech stacks; focus on delivering value.
•	Metadata quality significantly impacts chatbot accuracy.
From Collaboration:
•	Clear, structured communication accelerates alignment.
•	Iterative feedback loops with stakeholders prevent scope drift.
•	Balancing client needs with technical feasibility is crucial.
 
8. Future Recommendations
•	Add multilingual capabilities.
•	Enhance analytics to track user difficulties and content gaps.
•	Integrate intent classification for better permit narrowing.
•	Expand coverage beyond pilot agencies.
•	Continuous prompt refinement for improved accuracy.
 
9. Conclusion
The EEA Permit Navigator AI Chatbot demonstrates how AI + human-centered design can modernize government services. By automating routine navigation tasks, it improves accessibility, reduces staff workload, and sets a scalable foundation for future AI-driven public service tools.
Final Takeaway: The project not only delivers a functional, scalable MVP but also establishes a replicable model for deploying AI chatbots across state agencies—transforming how constituents interact with public information systems.
